当前 ZipShip 已经具备“上传静态产物 → 安全解压与检测 → 生成预览 → 发布 → 回滚”的功能型 MVP，领域建模、权限矩阵、版本模型和测试基础较好；主要缺口集中在生产构建、部署编排、发布一致性、后台任务、访问平面和未闭环的用户流程。

## 2. 严重级别

- **P0 / 上线阻塞**：会导致无法构建、无法部署、线上状态不一致、安全注销失效或核心链路不可用。
- **P1 / 可靠性与安全**：受控内测可能工作，但在并发、失败恢复、长期运行或公网环境下风险明显。
- **P2 / 产品完整度与维护性**：不阻塞最小 Web v1，但影响用户体验、扩展能力或维护成本。

## 3. P0：上线阻塞问题

### P0-01 API 生产构建失败

**现状**

- API 直接依赖 `playwright`，运行时检查代码在 API 入口静态导入。
- `bun build` 打包 Playwright Core 时无法解析 `chromium-bidi/lib/cjs/...`。
- Turbo 的 `typecheck` 依赖上游 `build`，因此根类型检查流水线也被构建错误打断。

**影响**

- CI build job 和生产 Dockerfile 无法形成可信的 API 镜像。
- 即使绕过打包，当前 API 镜像也没有安装 Chromium 浏览器；启用运行时检测仍可能在运行时失败。

**第二阶段处理原则**

- 浏览器运行时检查不得静态打进控制面主进程。
- 将检查器放进独立 Worker/可选容器，控制面只记录任务与结果。
- Rust 主服务必须能在没有浏览器依赖时完整构建和运行。

### P0-02 Docker 生产编排不是可运行的完整栈

**现状**

- Compose 启动 PostgreSQL、API、Nginx，但没有构建或挂载 Web Console 到 `/var/www/console`。
- 数据库迁移要求人工在容器外执行，没有 migration/init job。
- API 配置 `ZIPSHIP_STORAGE_ROOT=/data/sites`，代码会再追加 `sites`，实际站点落在 `/data/sites/sites/...`；Nginx 却从 `/data/sites/...` 查找。
- Nginx 只在 `/_console/` 暴露 Console；Vite 构建资源使用 `/assets/...`，React Router 使用 `/app`，路径基准不一致。
- PostgreSQL 暴露固定默认用户名和密码，生产模式没有 secret 注入和 fail-fast。

**影响**

- `infra:up` 无法真正完成“打开 Console → 上传 → 发布 → 从正式 URL 访问”的 smoke 流程。
- 即使手工复制 Console，静态资源、路由刷新和站点目录仍可能 404。

**关闭标准**

- 一条命令完成迁移、API、Worker、Console、访问平面和数据库启动。
- 在干净 Linux 主机上通过端到端 smoke：注册、创建项目、上传、预览、发布、回滚、重启后恢复。
- 所有生产密码/密钥必须从 secret/env 注入，生产环境缺少必填项时拒绝启动。

### P0-03 发布切换缺少跨数据库与存储的一致性

**现状**

- 发布/回滚先切换文件系统 `current` 链接，再执行数据库事务。
- 数据库失败后，访问平面可能已经指向新版本，而数据库仍记录旧版本。
- 当前链接替换过程是“创建临时链接 → 删除 current → rename”，删除与重命名之间存在空窗，并非严格原子。
- 部署记录固定写入 `success`，缺少 `pending/failed` 的真实状态与补偿信息。

**影响**

- 发布可能出现短暂 404。
- 控制台展示的线上版本与真实流量版本可能不一致。
- 失败后缺少自动补偿、人工恢复依据和幂等重试能力。

**第二阶段处理原则**

- 不再用软链接作为业务事实源；线上版本指针以 PostgreSQL 事务为唯一事实源。
- 产物以完整 SHA-256 内容寻址、不可变存储；发布只切换数据库指针。
- 访问平面通过内存缓存读取活动版本，事务提交后失效/更新缓存。
- 所有发布命令携带幂等键，并记录完整状态机和失败原因。

**Rust 第二阶段进度**

- `project_active_releases` 已成为正式访问版本的唯一事实源；发布/回滚不创建、删除或切换文件系统软链接。
- Project 行锁、Release/Artifact 共享锁、同项目复合外键和事务内 Upsert 保证并发发布严格排序；活动指针、成功 Deployment 与 Audit 要么全部提交，要么全部回滚。
- 相同幂等键和相同命令返回原 Deployment，不重复切换或写审计；相同键用于不同动作/目标返回稳定冲突。
- 真实 E2E 已证明数据库指针与 Access Plane 在 A → B → A 发布/回滚序列中一致。本问题已关闭。

### P0-04 Web 退出登录没有撤销服务端会话

**现状**

- 后端存在 logout 接口和 `revokedAt`。
- Console 的 logout 只删除 `sessionStorage`，没有调用服务端 logout。

**影响**

- 被复制或泄漏的 Refresh Token 在 7 天内仍然有效。
- 用户看到“已退出”，服务端却没有完成安全注销。

**关闭标准**

- Web logout 先请求服务端撤销，再清理本地状态；网络失败时明确提示并执行安全降级。
- Web 会话改为 `HttpOnly + Secure + SameSite` Cookie，配套 CSRF 策略。
- API Token 与 Web Session 使用不同凭证模型。

**Rust 第二阶段进度**

- Rust API 已实现服务端会话撤销、Cookie Session 和 CSRF，后端部分已关闭。
- 此问题保持“进行中”，直到 Console 删除旧 `sessionStorage` Bearer 流程并接入新 logout。

### P0-05 密码重置邮件指向不存在的前端路由

**现状**

- 后端可以创建、校验一次性密码重置 Token。
- 邮件链接是 `/reset-password/:token`，Console Router 没有该页面。
- 登录页没有“忘记密码”入口。

**影响**

- 用户收到邮件也无法完成密码重置，属于已暴露但不可用的流程。

**关闭标准**

- 请求、邮件、确认页面、成功后会话撤销、错误与过期状态全部贯通。
- 改密后撤销该用户所有既有 Web Session；是否撤销 API Token 作为显式产品策略。

## 4. P1：可靠性、安全与长期运行问题

### P1-01 上传和产物处理同步占用 HTTP 请求

- `/complete` 请求内同步执行解压、检测、Manifest、文件复制和可选浏览器检查。
- 大文件会长时间占用连接；代理超时或服务重启后没有可恢复执行上下文。
- 当前上传以 multipart `File` 接收，最大 512 MB，没有流式背压、分块、断点续传和取消。

第二阶段应使用持久化 Job：`queued → claimed → running → succeeded/failed/cancelled`，具备 lease、heartbeat、attempts、退避、启动恢复和超时回收。

**Rust 第二阶段进度**

- 上传正文已经直接从 Axum Body 流式写入隔离 staging，不进入整包内存；声明大小、服务上限和实际落盘字节必须一致。
- 客户端中断、正文过短或过长时删除 `.part` 文件并释放 transfer lease，同一个上传可以安全重传。
- `/complete` 不再执行解压和检测；它在 PostgreSQL 事务内创建 processing Release 与持久化 `artifact.process` Job，并立即返回 `202 Accepted`。
- 独立 Worker 已完成安全解压、Manifest、完整 SHA-256、不可变 Artifact 提交和 ready/failed 状态收敛；HTTP 请求不再承担产物处理。
- 本问题的“同步占用 HTTP”核心缺陷已关闭。上传状态可通过详情接口查询；字节级进度事件、分块断点续传和取消仍作为后续产品能力保留。

### P1-02 任务状态机缺少并发控制和恢复语义

- Upload Task 与 Release 有状态字段，但没有 Worker claim、锁定者、锁过期时间、重试次数和下一次执行时间。
- 相同任务重复 complete、服务崩溃、Worker 重启和并发发布的幂等边界不完整。
- 缺少取消和人工重试接口。

**Rust 第二阶段进度**

- 上传接收已具备数据库 transfer ID、租约到期时间和过期租约接管；重复 `/complete` 返回同一个 Release/Job，不重复写审计。
- 通用 Job 表和领域状态机已有 attempts、max_attempts、next_run_at、locked_by、locked_until、heartbeat_at 等字段。
- PostgreSQL Worker 已实现 `FOR UPDATE SKIP LOCKED` claim、owner/lease 校验、heartbeat、指数退避、最大尝试次数和过期 lease sweep；并发 Worker 不会重复领取有效任务。
- 成功、内容复用、可重试失败和终态失败均在单事务内收敛领域状态；进程退出会等待当前任务，异常中断后的任务由 lease 回收。
- 本问题的并发控制与崩溃恢复缺口已关闭。取消与人工重试 API 尚未实现，作为后续运维能力单独跟踪。

### P1-03 内容寻址目录可能被重复上传覆盖

- Release 使用 12 位短 Hash 作为目录名。
- 相同产物再次上传时会删除并重新复制目标目录；若该目录正在被线上版本引用，可能产生访问空窗。
- 短 Hash 冲突没有数据库唯一约束和完整 Hash 二次校验策略。

第二阶段应使用完整 SHA-256 作为 Artifact Blob 主键，先写临时目录并校验，再以原子 rename 完成只写一次的提交；Release 只引用 Artifact。

**Rust 第二阶段进度**

- Artifact 使用完整 SHA-256 内容寻址和分片目录；临时工作目录校验完成后才提交，已存在目标不会被删除或覆盖。
- 重复内容仅在完整 Hash、Manifest、文件数、总字节数和入口目录全部一致时复用；Release 只引用 Artifact 记录。
- 本问题已关闭。引用计数、保留策略和 GC 属于 P1-04，仍未完成。

### P1-04 项目和版本删除不清理存储

- 删除项目只删除数据库记录，磁盘上的上传包、临时目录和站点产物会遗留。
- Release 删除/归档接口未完成，Console 仍提示 Coming Soon。
- 没有引用计数、保留策略、垃圾回收任务、磁盘配额和低空间保护。

### P1-05 访问平面策略没有真正应用

- SPA fallback、缓存策略和自定义域名可以保存，并能生成 Nginx 片段预览。
- 没有服务把项目策略可靠写入配置并 reload。
- `projects.conf` 默认只读挂载且为空。
- 自定义域名没有所有权验证、DNS 状态、证书签发和失败状态机。

**Rust 第二阶段进度**

- 固定 Release Preview 已由 Rust Access Plane 提供，URL 绑定 Release UUID，不复用短 Artifact Hash。
- Access Plane 只查询 ready Release/Artifact，并按 Manifest 精确解析文件；SPA fallback 和 standard/aggressive 缓存策略已真正应用。
- 支持 MIME、强 ETag、`If-None-Match`、`If-Range`、单字节 Range、HEAD、`nosniff` 和 `no-referrer`；无效路径及未登记资源不会触碰文件系统。
- 控制面与用户产物使用强制不同的监听地址，默认分别为 `5006` 与 `5007`。
- 正式地址 `/{project_slug}/` 已根据数据库活动版本指针解析不可变 Artifact；未发布项目返回 404，发布/回滚提交后下一次请求立即解析新指针。
- 固定版本与活动版本访问策略部分已关闭；Caddy、自定义域名所有权验证与 TLS 仍在后续阶段，因此 P1-05 整体保持“进行中”。

### P1-06 Web Session 保存在可被脚本读取的 sessionStorage

- XSS 可以直接读取 Refresh Token。
- Web Nginx 没有完整 CSP；Tauri 配置中 CSP 为 `null`。
- 当前凭证既承担浏览器会话，又作为资源 API Bearer Token 使用，边界过宽。

**Rust 第二阶段进度**

- 新 Web Session 只在数据库保存 SHA-256 摘要；浏览器原始凭证通过 HttpOnly Cookie 承载，CSRF 使用独立令牌。
- API Token 已在新数据库中使用独立表和 scope/expiry 字段，但管理与鉴权 API 尚未实现。
- 此问题保持“进行中”，直到 Console 切换和 CSP 加固完成。

### P1-07 API Token 没有 scope、过期和配额

- API Token 一旦创建即继承用户全部权限，只能整体撤销。
- 没有 scope、expiresAt、IP/来源限制、使用审计细分和速率限制。
- 适合作为 MVP PAT，不适合作为成熟 CI/CD 凭证。

### P1-08 Webhook 缺少 URL 安全校验和可靠投递

- URL 只校验非空，没有限定 `https/http`、阻止回环/内网/云元数据地址，存在 SSRF 风险。
- 发布请求等待所有 Webhook 请求完成；无独立投递队列、超时、重试、签名版本、投递历史和死信状态。
- Secret 以明文存储。

### P1-09 生产可观测性与备份不足

- 有 JSON logger 和健康检查，但缺少统一 request/trace ID、访问日志、Prometheus 指标和任务指标。
- 没有 PostgreSQL 备份/恢复脚本和恢复演练。
- 没有磁盘容量、失败任务、发布延迟、Webhook 失败率告警。

### P1-10 配置在 production 下仍允许开发默认值

- 数据库地址、存储根、应用 URL、SMTP From 等均有开发默认值。
- `NODE_ENV=production` 不会自动要求显式配置。
- CORS 只允许 localhost/127.0.0.1；当 Console 与 API 分域时无法配置合法生产源。

### P1-11 TypeScript API 类型耦合阻碍 Rust 后端替换

- `@zipship/api-client` 和 Console 通过 Eden Treaty 直接依赖 `@zipship/api` 类型。
- Rust 服务替换 Elysia 后，这条类型链不能继续作为契约来源。

第二阶段应以 OpenAPI 为跨语言契约，生成 TypeScript Client，并用契约测试防止前后端漂移。

## 5. P2：产品完整度与维护性问题

### P2-01 Console 功能缺口

- Logs 页面仍是占位页；项目内虽然有 Activity Tab，但缺少组织级搜索、过滤和导出。
- Storage 页面仍是占位页；没有空间用量、Artifact 引用、清理和配额界面。
- API Token、Webhook、密码重置没有对应完整 UI。
- 缺少 Release 删除/归档、失败任务重试、上传取消、邀请管理入口的完整体验。

### P2-02 邮箱验证未实现

- 注册后立即获得完整会话。
- 没有邮箱所有权验证、重发验证邮件和未验证账号限制策略。

### P2-03 Desktop 仍是模板级薄壳

- Rust 端只有模板 `greet` 命令。
- Web 与 Desktop Runtime 的 `openExternal` 实现相同，没有调用 Tauri opener。
- 没有系统文件选择、路径直传、系统凭证库、设备授权、单实例和更新机制。
- 数据库虽然有 Desktop 登录相关表，但服务端没有完整路由与流程。

### P2-04 前端包体和加载策略

- Web/Desktop 主 JS 约 925 KB。
- Router 没有懒加载，页面和组件大多进入首屏包。
- lint 有 React Hooks 依赖警告，需在重构 API Client 时一并清理。

### P2-05 StorageAdapter 仅定义，未成为真实依赖边界

- 上传、处理、预览和发布仍直接调用本地文件系统函数。
- `LocalStorageAdapter` 没有被核心服务统一注入。
- S3/MinIO 不能复用当前软链接模型。

### P2-06 文档与当前实现漂移

- README 的端口说明与实际脚本不一致。
- TODO 顶部记录的测试数量和“构建全绿”已不是当前事实。
- `docs/04-TODO.md` 仍保留第一阶段早期状态。

## 7. 作为 v1 发布门槛的最小集合

以下项目全部满足后，才建议标记为生产可用 v1：

1. 干净 Linux 环境 `docker compose up` 后自动完成迁移并健康启动。
2. Rust API、Worker、Access Plane 和 Web Console 均有可复现构建。
3. 发布/回滚并发测试证明数据库活动版本与真实访问版本一致。

5. 登录、服务端注销、密码重置、会话过期完整贯通。
6. ZIP 路径安全、压缩炸弹、超限上传和静态路径穿越测试全部通过。
7. 项目删除、Release 清理和 Artifact GC 不会误删活动或仍被引用的内容。
8. PostgreSQL 备份与恢复至少完成一次自动化演练。
9. 全量 unit、integration、contract、E2E 和生产 smoke 在 CI 中通过。
