# Product

## Register

product

## Users

- **独立开发者 / 小团队**：自建服务器部署前端构建产物，需要一个轻量、可靠、可控的部署工具，不想依赖 Vercel / Netlify 等第三方平台。
- **DevOps / 运维人员**：管理多个静态项目的版本、发布与回滚，需要清晰的版本历史和审计跟踪。
- **内部工具团队**：将构建好的前端产物上传到内部服务器，给测试或产品团队预览。

用户的使用场景是**日常操作**——登录、查看项目、上传产物、检查检测结果、发布、回滚。他们需要快速完成这些操作，不被复杂界面干扰。

## Product Purpose

ZipShip 是一个面向静态产物的私有化部署工具。用户上传已构建的静态产物（zip），平台负责解压、安全检测、内容寻址版本化、生成测试地址、发布正式版本和快速回滚。

它不是 Jenkins 那样的 CI/CD 平台，也不是 Vercel 那样的源码构建平台。它的核心价值是：**把「上传 → 检测 → 测试 → 发布 → 回滚」这条链路做到极致稳定、清晰、快速**。

成功标准：用户打开 Console 后，能在 3 步内完成一次发布，在 2 步内完成一次回滚，任何时候都清楚当前哪个版本在线上。

## Brand Personality

**精准 · 自信 · 清爽**

- **精准**：每个操作都有明确的反馈，不模糊、不拖沓。界面信息密度恰到好处，不冗余不缺失。
- **自信**：关乎部署的工具必须传达可靠感。稳定、可预期、不出错。用户信任这个工具来发布他们的产品。
- **清爽**：消费级产品的外观，不是企业后台。留白充足、视觉层次干净、没有杂乱的功能堆砌。

## Anti-references

- **不要「AI 工具感」**：不使用蓝色、深蓝、蓝紫色系。不出现渐变文字、毛玻璃默认态、过度圆角（32px+ 卡片）等 AI 工具常见视觉特征。
- **不要企业后台感**：不像 Jenkins、Grafana、Kibana 那种信息密度极高、视觉沉重的管理系统。不堆砌卡片网格。
- **不要 SaaS 模板感**：不套用 Big number + small label 的 hero-metric 模板，不每段都用小字号全大写 eyebrow 标题。

## Design Principles

1. **操作即反馈** — ZipShip 的核心动作（上传、发布、回滚）都有即时、清晰的视觉反馈。用户从不怀疑操作是否生效。Toast、进度、状态标签都是反馈系统的一部分，不是装饰。

2. **消费级精致感** — 虽然是部署工具，但体验对标消费级产品。留白、动效、过渡、文案都在传达「这个工具值得被信任」。不是「能用就行」，而是「用得舒服」。

3. **信息层次分明** — 每个页面只有一个核心任务。次要信息折叠或降级，不干扰主要操作流。版本列表、检测结果、发布按钮的视觉权重经过设计，不是平铺。

4. **一致性即可信度** — 组件的表现是可预测的：同一个操作在页面任何位置行为一致；同一个状态（active / ready / failed）颜色语义一致；点击、悬停、过渡节奏一致。用户建立起心智模型后就无需重新学习。

5. **深色模式不是事后补丁** — Day / Night 两种主题从设计系统层面共存，而不是浅色做完再「适配」深色。两种主题同等精致。

## Accessibility & Inclusion

- 目标 WCAG 2.1 AA。
- 支持中英文界面，文案不硬编码。
- 考虑前端的 contrast ratio（- 非满足标题级要求后再降级）、focus indicator、keyboard navigation。
- 所有动效考虑 `prefers-reduced-motion`，提供减少动画的替代方案。
- 颜色不依赖色相作为唯一信息传递手段（状态标签同时使用文字 + 颜色）。
